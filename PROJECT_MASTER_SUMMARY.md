# Цель проекта

Определить, существует ли достаточно большой рынок компаний, испытывающих регулярные проблемы с учетом дорогостоящих мобильных активов.

До подтверждения рынка разработка продукта не начинается.

---

# Главная гипотеза

Компании теряют деньги не из-за отсутствия складского учета.

Основная проблема — отсутствие контроля над дорогостоящими мобильными активами, которые постоянно перемещаются между объектами и сотрудниками.

Типичные вопросы бизнеса:

* Где сейчас находится оборудование?
* Кто его получил?
* На каком объекте оно используется?
* Когда оно должно вернуться?
* Кто несет ответственность за потерю?
* Когда заканчивается поверка оборудования?

---

# Текущий ICP

Компания соответствует ICP, если выполняется большинство условий:

* собственный штат сотрудников;
* несколько мобильных бригад;
* дорогостоящее оборудование;
* оборудование регулярно перемещается между объектами;
* потеря оборудования ощутима финансово;
* существует персональная ответственность за оборудование.

Наиболее перспективные направления:

* промышленный электромонтаж;
* энергетика;
* электролаборатории;
* геодезия;
* инженерные сети;
* обслуживание промышленной инфраструктуры;
* сервис тяжелого оборудования.

Подробнее — `docs/ICP.md`.

# Главный вывод

Главный фактор — не отрасль.

Главный фактор — наличие дорогостоящих мобильных активов, регулярно перемещающихся между людьми и объектами.

Именно вокруг этой проблемы строится Customer Discovery.

# Текущие ограничения

До завершения Customer Discovery запрещается:

* проектировать SaaS;
* писать MVP;
* разрабатывать архитектуру продукта;
* принимать решения, основанные только на предположениях.

Все решения должны опираться на результаты исследования.

# Продолжение работы в новом чате

## Модель взаимодействия

Во время работы над проектом агент выполняет роль технического аналитика и архитектора.

Пользователь выполняет роль Product Owner.

### Задачи агента

* анализировать текущую ситуацию;
* предлагать ровно один следующий небольшой, логически обоснованный шаг;
* объяснять, почему именно этот шаг является следующим;
* не предлагать несколько альтернативных планов без необходимости;
* не забегать на несколько этапов вперед.

### Задачи Product Owner

* принимать или отклонять предложенный шаг;
* корректировать направление проекта;
* выполнять действия, которые агент не может выполнить самостоятельно;
* предоставлять результаты для дальнейшего анализа.

Работа строится как последовательность небольших итераций.
