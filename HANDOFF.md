# HANDOFF

> Оперативная память проекта. Передает только информацию, необходимую для продолжения работы и не дублирует постоянную документацию.

---

# 1. Текущее состояние

Проект находится на стадии Customer Discovery.

В течение текущей сессии была существенно улучшена инженерная инфраструктура проекта: сформирована система передачи контекста между рабочими сессиями, разработан шаблон HANDOFF и подтверждена его работоспособность в ходе независимого тестирования с использованием Codex.

Основная инженерная задача по развитию Validation Framework остается незавершенной и будет продолжена в следующей сессии.

---

# 2. Что выполнено

* Реализован фундамент Validation Framework.
* Реализованы Parser и Validator для `schema.sql`.
* Создан единый вход `tools/validate.py`.
* Переработан `PROJECT_MASTER_SUMMARY.md`.
* Создан `templates/HANDOFF.template.md`.
* Проведен первый независимый эксперимент передачи проекта новому агенту (Codex).
* На основании результатов эксперимента разработана версия `HANDOFF.template.md` v1.1.
* Сформулирован и зафиксирован принцип эмпирического развития инженерного процесса.

---

# 3. Принятые решения

* Документация репозитория остается единственным источником истины.
* `HANDOFF.md` является оперативной памятью проекта.
* HANDOFF всегда создается агентом, участвовавшим в рабочей сессии.
* Передача проекта новому агенту осуществляется по цепочке:

  * `PROJECT_MASTER_SUMMARY.md`;
  * `HANDOFF.md`;
  * профильная документация.
* Шаблоны и инженерные процессы развиваются только после практических экспериментов.
* Основной принцип развития процесса:
  **«Сначала эксперимент. Потом изменение процесса.»**

---

# 4. Текущая активная задача

Развитие Validation Framework.

Цель — перейти от проверки отдельных артефактов к автоматической проверке согласованности документации проекта.

---

# 5. Где остановились

Реализованы:

* Parser для `schema.sql`;
* Validator структуры схемы;
* Documentation Validator;
* единый запуск Validation Framework.

Следующая вертикаль еще не начата:

* Parser для `DOMAIN_MODEL.md`;
* Validator согласованности между `DOMAIN_MODEL.md` и `schema.sql`.

Во время текущей сессии внимание было временно переключено на улучшение инженерной инфраструктуры проекта.

---

# 6. Следующий инженерный шаг

Реализовать `tools/parsers/domain_model.py`, используя архитектуру, уже примененную в `schema.py`.

После реализации убедиться, что парсер возвращает перечень сущностей доменной модели в удобном для валидаторов формате.

---

# 7. Что делать после завершения задачи

После реализации парсера приступить к созданию первого Validator согласованности между доменной моделью и физической схемой базы данных.

После завершения этой вертикали вернуться к развитию Customer Discovery и накоплению исследовательской базы компаний.

---

# 8. Что не пересматривать

* Структуру проектной документации.
* Общую архитектуру Validation Framework.
* Использование документации как единственного источника истины.
* Архитектуру передачи контекста через `PROJECT_MASTER_SUMMARY.md` и `HANDOFF.md`.
* Принцип эмпирического развития инженерного процесса.

---

# 9. Дополнительный контекст

Первый независимый тест с использованием Codex признан успешным.

Эксперимент показал, что новый агент способен корректно восстановить контекст проекта, самостоятельно выбрать необходимую документацию и предложить дальнейшие действия.

При этом эксперимент выявил важный недостаток первой версии HANDOFF: новый агент понимал стратегическую цель проекта, но не понимал текущую инженерную задачу. На основании этого была разработана версия шаблона HANDOFF v1.1, вводящая разделы «Текущая активная задача», «Следующий инженерный шаг» и «Что делать после завершения задачи».

Следующим шагом после обновления шаблона должен стать повторный независимый прогон Codex для проверки, устранили ли изменения выявленный недостаток.
