# Структура документации проекта

Данный документ определяет назначение каждого документа в репозитории.

## Главный принцип

> Каждый факт должен иметь только один источник истины (Single Source of Truth).

Документы должны дополнять друг друга, а не дублировать информацию.

---

# Структура репозитория

```text
/
├── README.md
├── PROJECT_MASTER_SUMMARY.md
│
├── docs/

├── research/

```

# Документы корня репозитория

## README.md

### Назначение

Точка входа в репозиторий.

### Содержит

- краткое описание проекта;
- цель проекта;
- текущую стадию;
- структуру репозитория;
- ссылки на основную документацию.

### Не содержит

- результаты исследований;
- гипотезы;
- рабочие заметки.

---

## PROJECT_MASTER_SUMMARY.md

### Назначение

Главный документ проекта.

Это единственный источник информации о текущем состоянии проекта.

### Содержит

- цель проекта;
- текущую стадию;
- подтвержденные выводы;
- опровергнутые гипотезы;
- актуальный ICP;
- текущие ограничения;
- направление развития проекта.

### Обновляется

Только при существенном изменении проекта.

### Не содержит

- историю исследований;
- ежедневные заметки;
- технические детали реализации.

- процесс Discovery;

---

## ICP.md

### Назначение

Описание идеального клиента.

### Содержит

- обязательные признаки;
- желательные признаки;
- Anti-ICP;
- критерии квалификации;
- историю изменения ICP.

### Не содержит

- описание конкретных компаний.

---

## HYPOTHESES.md

### Назначение

Реестр всех гипотез проекта.

Каждая гипотеза имеет статус:

- Proposed
- Testing
- Confirmed
- Rejected

Для каждой гипотезы фиксируются:

- формулировка;
- причина появления;
- способ проверки;
- результат проверки.

---

## DECISIONS.md

### Назначение

Журнал принятых решений (Architecture Decision Record).

Каждая запись содержит:

- дату;
- проблему;
- принятое решение;
- обоснование;
- последствия.

Решения не пересматриваются без появления новых фактов.

---

## RESEARCH_LOG.md

### Назначение

Хронологический журнал исследований.

Каждая запись содержит:

- дату;
- исследуемую компанию;
- наблюдения;
- выводы;
- какие гипотезы подтверждены или опровергнуты.

Это журнал процесса исследования, а не итоговых выводов.

---

## QUESTIONS.md

### Назначение

Список открытых вопросов проекта.

Каждый вопрос имеет статус:

- Open
- In Progress
- Answered
- Closed

После получения ответа вопрос должен:

- быть закрыт;
- превратиться в гипотезу;
- превратиться в принятое решение;
- привести к изменению ICP.

---

## GLOSSARY.md

### Назначение

Единый словарь терминов проекта.

### Содержит

* термины предметной области;

Каждый файл содержит:

# Общие правила

## Один факт — одно место

Каждая информация хранится только в одном документе.

---

## Без дублирования

Документы должны ссылаться друг на друга, а не копировать информацию.

---

## Факты важнее предположений

Наблюдения и выводы должны быть разделены.

Любое предположение считается гипотезой до момента проверки.

---

## Живая документация

Документация отражает текущее состояние проекта и развивается вместе с ним.

Она не является архивом переписки, а служит рабочим инструментом принятия решений.

## data/schema.sql

### Назначение

Единственный источник истины для структуры базы данных проекта.

### Содержит

* описание таблиц;
* ограничения;
* связи между таблицами;
* начальные справочные данные (seed data).

### Не содержит

* рабочие данные;
* результаты исследований;
* бизнес-логику;
* SQL-запросы приложения.

База данных `discovery.db` должна создаваться из `schema.sql`.

## data/discovery.db

### Назначение

Рабочая база данных проекта.

Создается автоматически на основе `schema.sql`.

### Содержит

* исследуемые компании;
* наблюдения;
* доказательства;
* контакты;
* интервью;
* задачи;
* рабочие данные проекта.

### Не содержит

* структуру базы данных;
* документацию;
* исходный код.

### Правила

* `discovery.db` не является источником истины.
* Структура базы определяется только `schema.sql`.
* Файл `discovery.db` не хранится в Git и добавляется в `.gitignore`.


## collector/

### Назначение

Исходный код инструментов автоматизированного сбора данных.

### Содержит

* парсеры сайтов;
* коллекторы компаний;
* импорт данных;
* экспорт данных;
* вспомогательные утилиты.

### Не содержит

* бизнес-логику проекта;
* описание предметной области;
* результаты исследований.

### Правила

Collector отвечает только за получение и сохранение данных.

Любая логика анализа, оценки ICP и принятия решений должна находиться вне Collector.


## research/companies/

### Назначение

База знаний по исследуемым компаниям.

Одна компания — один Markdown-файл.

### Содержит

* общую информацию о компании;
* ссылки на сайт;
* краткое описание бизнеса;
* результаты исследования;
* ссылки на связанные записи в базе данных;
* выводы исследователя.

### Не содержит

* справочники;
* бизнес-логику;
* структуру базы данных.

### Правила

Markdown-файл представляет собой человекочитаемое исследовательское досье.

Рабочие данные (наблюдения, контакты, интервью, задачи) хранятся в `discovery.db`.

Файл компании может ссылаться на записи в базе данных, но не должен их дублировать.


## research/reports/

### Назначение

Хранение аналитических отчетов, подготовленных на основе результатов исследований.

### Содержит

* обзоры рынка;
* анализ сегментов;
* сравнение ICP;
* сводные отчеты;
* промежуточные результаты исследований.

### Не содержит

* сырые данные;
* рабочие записи;
* структуру базы данных.

### Правила

Каждый отчет является аналитическим документом, основанным на данных из `discovery.db` и исследованиях компаний.

Отчеты могут содержать выводы и интерпретации, но не являются источником первичных данных.


## research/screenshots/

### Назначение

Хранение визуальных доказательств, полученных в ходе исследования компаний.

### Содержит

* скриншоты сайтов;
* скриншоты вакансий;
* скриншоты социальных сетей;
* изображения, подтверждающие наблюдения.

### Не содержит

* аналитические выводы;
* описание компаний;
* структуру базы данных.

### Правила

Каждый скриншот является потенциальным доказательством (`Evidence`) и при необходимости должен быть связан с соответствующим Observation в базе данных.

Скриншоты используются как архив первоисточников и не заменяют текстовое описание наблюдений.

## research/exports/

### Назначение

Хранение данных, подготовленных для передачи во внешние системы или анализа сторонними инструментами.

### Содержит

* CSV-экспорты;
* Excel-файлы;
* JSON-выгрузки;
* промежуточные наборы данных.

### Не содержит

* исходные данные;
* документацию проекта;
* бизнес-логику.

### Правила

Все файлы в `exports` являются производными от данных проекта.

Источником истины остаются `discovery.db`, `schema.sql` и документация проекта.

## data/backups/

### Назначение

Хранение резервных копий рабочей базы данных.

### Содержит

* архивные копии `discovery.db`;
* снимки базы перед существенными изменениями;
* резервные копии для восстановления.

### Не содержит

* документацию;
* исходный код;
* результаты анализа.

### Правила

Резервные копии используются только для восстановления данных.

Источником структуры базы данных остается `schema.sql`.

Резервные копии не хранятся в Git.

# Иерархия источников истины

При возникновении противоречий документы имеют следующий приоритет:

1. `PROJECT_MASTER_SUMMARY.md` — текущее состояние проекта.
2. `WORKFLOW.md` — правила ведения проекта.
3. `DOMAIN_MODEL.md` — предметная область и бизнес-модель.
4. `DATA_MODEL.md` — логическая модель данных.
5. `schema.sql` — физическая модель данных.
6. `discovery.db` — рабочие данные проекта.
7. Остальная документация.

Изменение документа более высокого уровня должно сопровождаться актуализацией всех зависимых документов.

# Эволюция документации

Документация является частью архитектуры проекта и развивается вместе с ним.

При добавлении новой сущности, процесса или компонента необходимо:

1. Определить, существует ли подходящий документ.
2. Если нет — создать новый документ с четко определенной ответственностью.
3. Не дублировать информацию между документами.
4. Обновить `00_DOCUMENT_STRUCTURE.md`.
5. При необходимости обновить `PROJECT_MASTER_SUMMARY.md`.

Каждый новый документ должен иметь:

* назначение;
* область ответственности;
* описание содержимого;
* ограничения (что документ не содержит).

Структура документации должна оставаться простой, непротиворечивой и соответствовать принципу **Single Source of Truth**.
