# ARCHITECTURE

## Цель

## Архитектурные принципы

* Single Source of Truth — каждый факт имеет единственный источник истины.
* Separation of Concerns — каждый компонент имеет одну четко определенную ответственность.
* Domain First — предметная область определяет структуру данных, а не наоборот.
* Documentation First — архитектурные решения фиксируются в документации до реализации.
* Validation First — согласованность проекта должна проверяться автоматически.
* Evolutionary Architecture — архитектура развивается постепенно и фиксируется через ADR.
* Human-readable by Default — документация должна оставаться понятной человеку без изучения исходного кода.
* Code Follows Documentation — реализация должна соответствовать утвержденной документации.
* Layered Validation — подсистема Validation разделяется на три слоя:

  * Readers — чтение файлов;
  * Parsers — преобразование файлов в структурированные данные;
  * Validators — проверка согласованности структурированных данных.

Каждый слой отвечает только за свою задачу и не должен содержать ответственность другого слоя.


## Компоненты системы

### Documentation

Хранит знания о проекте, правила работы и архитектурные решения.

---

### Domain Model

Описывает предметную область, агрегаты, сущности, инварианты и бизнес-правила.

---

### Data Model

Описывает логическую модель данных и связи между сущностями.

---

### Database

Хранит структурированные данные проекта.

Структура определяется `schema.sql`.

Рабочие данные находятся в `discovery.db`.

---

### Collector

Автоматически собирает информацию из внешних источников и сохраняет ее в базу данных.

---

### Validation

Проверяет согласованность документации, доменной модели, модели данных и структуры базы данных.

---

### Research

Хранит результаты исследований, аналитические материалы и человекочитаемые отчеты.

### Описание

1. Collector получает данные из внешних источников.
2. Данные сохраняются в `discovery.db`.
3. Domain Model интерпретирует данные в рамках предметной области. Domain Model не лежит "после" базы данных. Она существует параллельно и определяет смысл данных. На диаграмме это упрощено как поток обработки, но позже можно будет сделать более точную архитектурную схему, где Domain Model станет центральным элементом между Collector, Database и приложением.
4. На основе доменной модели выполняется Customer Discovery и оценка соответствия ICP.
5. Результаты исследования оформляются в виде аналитических отчетов.



## Автоматическая проверка

Целостность архитектуры должна проверяться автоматически.

Планируемые валидаторы:

* `validate_repository.py` — проверка структуры репозитория.
* `validate_docs.py` — проверка документации.
* `validate_domain.py` — проверка доменной модели.
* `validate_data_model.py` — проверка модели данных.
* `validate_schema.py` — проверка соответствия `schema.sql` и доменной модели.

Все валидаторы должны запускаться перед коммитом и могут использоваться в GitHub Actions.

При невозможности автоматической проверки допускается ручная валидация согласно `WORKFLOW.md`.


## Будущие компоненты

Следующие компоненты не входят в текущую архитектуру, но могут появиться после завершения Customer Discovery:

* Web Application
* REST API
* Authentication
* User Management
* Background Jobs
* AI Assistant
* Notification Service
* Reporting Dashboard
* Integrations (CRM, ERP и др.)

Добавление новых компонентов допускается только после принятия соответствующего ADR и обновления архитектурной документации.

### Назначение

Подсистема Validation автоматически проверяет согласованность всех архитектурных артефактов проекта.

Validation не изменяет данные проекта и не принимает архитектурных решений. Ее задача — обнаруживать нарушения и сообщать о них.
