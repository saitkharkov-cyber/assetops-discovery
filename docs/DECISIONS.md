# DECISIONS

## Назначение

Документ фиксирует все важные решения проекта.

Цель документа — сохранить причины принятия решений и избежать повторного обсуждения уже закрытых вопросов.

Каждое решение должно быть основано на фактах, а не на предположениях.

---

# Правила

Каждое решение получает уникальный идентификатор.

Формат:

```
ADR-001
ADR-002
ADR-003
...
```

Каждое решение содержит:

* дату;
* проблему;
* принятое решение;
* обоснование;
* последствия.

---

# ADR-001

## Дата

2026-06-26

## Проблема

Существует соблазн начать разработку SaaS до проверки существования рынка.

## Решение

Не приступать к разработке продукта до завершения Customer Discovery.

## Обоснование

Рынок и ICP еще не подтверждены.

Разработка до проверки спроса создает высокий риск построить продукт для слишком узкой или несуществующей ниши.

## Последствия

Приоритет получают:

* исследование рынка;
* Customer Discovery;
* проверка гипотез;
* уточнение ICP.

# ADR-002

## Дата

2026-06-26

## Проблема

Первоначально предполагалось использовать автоматический скоринг компаний.

## Решение

На этапе Discovery использовать ручной анализ компаний.

## Обоснование

Первые попытки автоматического скоринга дали большое количество ложноположительных результатов.

Человек пока значительно лучше определяет инженерную сложность бизнеса и наличие потенциальной боли.

## Последствия

Автоматизация откладывается до накопления достаточной исследовательской базы.

# ADR-003

## Дата

2026-06-26

## Проблема

Необходимо определить единый подход к ведению проекта.

## Решение

Работать по принципу «один следующий шаг».

## Обоснование

Небольшие проверяемые итерации уменьшают когнитивную нагрузку, предотвращают преждевременное проектирование и позволяют быстрее корректировать направление проекта.

## Последствия

По команде **«дальше»** всегда выполняется только один следующий шаг.

# ADR-004

## Дата

2026-06-26

## Проблема

Необходимо организовать документацию проекта таким образом, чтобы информация не дублировалась.

## Решение

Каждый документ имеет единственную ответственность (Single Responsibility).

## Обоснование

Документы легче поддерживать, если каждый отвечает только за одну область знаний.

## Последствия

Структура документации фиксируется в:

`docs/00_DOCUMENT_STRUCTURE.md`

# Правила обновления

Новое решение добавляется только тогда, когда:

* принято важное техническое решение;
* изменен процесс работы;
* изменен подход к исследованию;
* изменена стратегия проекта.

Каждое решение должно иметь обоснование.

Если решение впоследствии изменяется, создается новая запись с новым идентификатором.

История решений не переписывается.
