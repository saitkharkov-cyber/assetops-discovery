# WORKFLOW

## Назначение

Данный документ определяет правила ведения проекта.

Все участники проекта придерживаются этих правил независимо от текущей стадии разработки.

---

# Главный принцип

> Цель проекта — найти истину, а не подтвердить первоначальную идею.

Любая гипотеза может быть подтверждена, изменена или полностью отвергнута.

---

# Принцип одного шага

Работа ведется итерациями.

Каждый следующий шаг выполняется только после завершения предыдущего.

Не допускается планирование нескольких шагов вперед, если текущий шаг еще не завершен.

---

# Discovery раньше разработки

До завершения Customer Discovery запрещается:

* проектировать архитектуру SaaS;
* писать MVP;
* реализовывать функциональность продукта;
* принимать технические решения на основе предположений.

Любая разработка начинается только после подтверждения существования рынка и повторяемой бизнес-боли.

---

# Evidence First

Любое утверждение должно относиться к одной из категорий:

* **Факт** — подтвержден наблюдением или исследованием.
* **Гипотеза** — требует проверки.
* **Предположение** — личная идея, не имеющая подтверждений.

При обсуждении необходимо четко разделять эти категории.

---

# Правило новых фактов

Каждый новый факт должен привести только к одному из результатов:

* подтверждает существующую гипотезу;
* опровергает существующую гипотезу;
* требует создания новой гипотезы.

Если факт не влияет ни на что, его не следует фиксировать.

---

# Документирование

Каждый документ имеет собственную ответственность.

Информация хранится только в одном месте.

Дублирование информации между документами запрещено.

Структура документации описана в:

`docs/00_DOCUMENT_STRUCTURE.md`

---

## Язык документации и идентификаторов

В проекте используется два языка с разными зонами ответственности.

### Русский язык

Используется для:

* документации;
* описания гипотез;
* результатов исследований;
* комментариев;
* бизнес-терминов.

Документация должна быть максимально понятной человеку.

---

### Английские идентификаторы (Code)

Используются исключительно как стабильные идентификаторы.

Применяются в:

* базе данных;
* SQL-схемах;
* названиях полей;
* коде;
* API;
* парсерах;
* внутренних ссылках между сущностями.

Идентификатор должен быть:

* коротким;
* уникальным;
* неизменяемым;
* написанным в `snake_case`.

Примеры:

| Русское название           | Code               |
| -------------------------- | ------------------ |
| Собственный штат           | `own_employees`    |
| Мобильные бригады          | `mobile_crews`     |
| Электролаборатория         | `electrical_lab`   |
| Дорогостоящее оборудование | `expensive_assets` |

---

### Правило

Русское название может изменяться для повышения понятности.

Английский `Code` после появления в базе данных считается постоянным и изменению не подлежит.

---

### Связь доменной модели и базы данных

Каждая сущность, агрегат или справочник, описанные в `DOMAIN_MODEL.md`, должны явно указывать соответствующую таблицу базы данных (если такая таблица существует).

Формат:

* `Aggregate: Company (table: companies)`
* `Entity: Observation (table: observations)`
* `Reference Data: ObservationType (table: observation_types)`

Это обеспечивает однозначное соответствие между доменной моделью и физической моделью данных.

---

# Принятие решений

Решения принимаются только после появления достаточного количества фактов.

После фиксации решение записывается в `DECISIONS.md`.

Повторное обсуждение решения допускается только при появлении новых данных.

---

# Исследование компаний

Каждая исследуемая компания оформляется отдельным Markdown-файлом.

Одна компания = один документ.

Исследование должно основываться на фактах, найденных на сайте компании или полученных в ходе CustDev.

---

# Обновление ICP

ICP изменяется только тогда, когда новые исследования показывают устойчивую закономерность.

ICP не меняется на основании одной компании.

---

# Управление гипотезами

Все гипотезы проходят жизненный цикл:

```text
Proposed
    ↓
Testing
    ↓
Confirmed
или
Rejected
```

Гипотеза не может находиться одновременно в нескольких состояниях.

---

# Работа с контекстом

При возобновлении проекта рекомендуется следующий порядок чтения:

1. `PROJECT_MASTER_SUMMARY.md`
2. `docs/WORKFLOW.md`
3. `docs/ICP.md`
4. `docs/DECISIONS.md`
5. Последние записи в `docs/RESEARCH_LOG.md`

---

# Основной критерий качества

Проект считается движущимся в правильном направлении, если каждое новое исследование уменьшает количество неопределенности.

Количество написанного кода не является показателем прогресса.

## Команда «дальше»

При взаимодействии с ассистентом команда **«дальше»** имеет специальное значение.

После получения команды **«дальше»** ассистент должен:

* предложить только один следующий шаг;
* не перечислять последующие шаги;
* не строить дорожную карту;
* не забегать вперед;
* дождаться завершения текущего шага.

Каждый шаг должен представлять собой одно законченное действие, которое можно выполнить, проверить и только после этого переходить к следующему.

Цель такого подхода — минимизировать когнитивную нагрузку, избежать преждевременного проектирования и двигаться небольшими проверяемыми итерациями.

Если для выполнения следующего шага требуется результат предыдущего, ассистент не должен предлагать альтернативные или параллельные действия.


## Синхронизация артефактов

При изменении проекта необходимо проверить, какие артефакты требуют обновления.

Минимальный чек-лист:

* `PROJECT_MASTER_SUMMARY.md` — изменилось ли текущее состояние проекта?
* `00_DOCUMENT_STRUCTURE.md` — появились ли новые документы или каталоги?
* `DOMAIN_MODEL.md` — изменилась ли предметная область?
* `DATA_MODEL.md` — изменилась ли логическая модель данных?
* `schema.sql` — изменилась ли физическая модель данных?
* `GLOSSARY.md` — появились ли новые термины?
* `DECISIONS.md` — было ли принято архитектурное решение?

Коммит выполняется только после проверки актуальности всех связанных артефактов.

При наличии автоматических валидаторов (tools/validate_*) они являются основным механизмом проверки согласованности проекта.

Ручная проверка используется только для тех аспектов, которые невозможно проверить автоматически.
